# 00 · System Contract

本文档是 Open Novel 技术体系的入口契约。它不堆完整版本表、代码片段或 schema,只说明系统如何分层、每层拥有什么主权,以及实现前必须通过哪些事实审计。

## 职责边界

本篇负责三件事:

- 定义技术路线的顶层承诺。
- 定义核心 spec 与 appendix 的分工。
- 定义代码实施前的审计闸门。

本篇不负责:

- 维护完整依赖版本表。
- 展开数据库表结构、事件枚举或 JSON Schema。
- 重复各运行时层的内部流程。

## 技术路线

Open Novel 是本地单机创作工作台。应用以一个单仓单应用承载前端、后端接口、Agent runner、存储访问与编辑器集成。Agent 运行时使用自定义 runner,不引入通用 Agent 框架;模型调用、结构化输出、工具执行、审批挂起与流式事件都由本仓库自己的契约连接。

系统数据分为三类:

- 项目事实:作品设定、章节索引、知识图谱、审批记录与派生索引。
- 运行时记忆:跨项目会话 thread、message 与压缩历史。
- 过程历史:单项目内的调用日志、trace、成本与调试事件。

这三类数据分库管理,避免作品事实、会话历史和调试过程互相污染。

## 输入、输出与依赖

输入是产品 PRD、设计文档、外部依赖事实和实施前审计结果。输出是系统分层边界、技术路线约束、审计是否阻塞的判断,以及 appendix 明细的归属规则。本篇依赖 README 的文档入口和 AGENTS/CLAUDE 的工作规范,下游约束全部核心 spec。

## Spec 写作契约

核心 spec 写系统契约,不当实现明细仓库。每篇核心 spec 必须讲清:

- 它解决什么问题。
- 它负责什么,不负责什么。
- 它拥有哪些主权对象。
- 关键流程如何流转。
- 失败时如何处理。
- 用户能看到什么结果。
- 哪些实现明细被后置到 appendix。

核心 spec 不堆完整表结构、完整 JSON Schema、大段代码、工具参数全量枚举、prompt 全文、测试矩阵或迁移细节。

如果一个细节变化不会改变职责边界、技术路径、用户体验承诺或失败语义,它不进入核心 spec 正文,而进入 appendix。

## Appendix 规则

appendix 负责承接实现明细,但不能变成无主信息堆。每个 appendix 条目都必须服务于某个核心契约,并能被核心 spec 追溯引用。

关键约束不能被完全挪走。如果某个字段、状态、事件或规则会直接影响系统行为,核心 spec 必须点名说明它的存在;完整定义、字段展开和样例放入 appendix。

## 审计闸门

代码实施前必须验证所有会影响运行路径的外部事实,包括模型能力、结构化输出行为、native binding、构建配置和运行时限制。审计产物回写 appendix,核心 spec 只记录审计是否阻塞系统路径。

审计失败时不静默降级到行为差异很大的技术路线。系统必须选择显式失败、用户确认或回写设计,而不是让实现绕开契约。

## 失败语义

系统级失败分三类:

- 事实未验证:不能开始对应实现,记录到 TODO。
- 依赖能力不满足:回写技术路线,不得让业务层绕行。
- 契约冲突:以核心 spec 的主权边界为准,重复定义的一方必须删除或改为引用。
