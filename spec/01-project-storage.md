# 01 · Project Storage

本文档定义项目事实的存储边界。它说明作品文件、项目索引与派生数据如何协作,不在正文展开完整表结构或迁移脚本。

## 职责边界

本篇负责:

- 项目文件与项目事实库的职责划分。
- 项目事实、派生索引、审批记录和知识图谱的主权边界。
- 外部编辑、派生文件、回滚和连接生命周期的设计原则。

本篇不负责:

- 跨项目会话记忆。
- 单项目过程日志。
- Agent prompt、工具参数或 UI 交互细节。
- 完整表结构和字段枚举。

## 主权对象

项目存储层拥有这些对象:

- 作品 Markdown 文件。
- 设定文件与文件 frontmatter。
- 项目事实索引。
- 派生索引与反向引用。
- 审批落盘记录。
- 项目级长期事实辅助数据。

项目存储层只保存“作品事实”和“可由作品事实推导的索引”。会话消息、模型调用日志、trace 与成本数据不属于本层。

## 输入、输出与依赖

输入是用户编辑、审批通过的变更、外部文件变更和 reindex 请求。输出是作品文件、项目事实、派生索引刷新结果和冲突状态。本篇依赖 turn orchestration 的落盘决策、knowledge graph 的索引需求和 runtime state 的分库边界。

## 技术路径

作者可读、可带走、可审查的作品内容保存在文件系统中。系统需要高频查询、级联分析或一致性校验的事实进入项目事实库。派生索引由后台 reindex 维护,不是作者直接编辑的真源。

写入路径遵循同一条链路:

1. Agent 或用户产生变更意图。
2. 需要落盘的变更进入审批或用户确认路径。
3. 存储层执行文件写入与事实库更新。
4. reindex 根据变更范围刷新派生索引。
5. 失败时回滚到可解释状态,并把受影响的审批标记为不可继续使用。

## 派生数据原则

派生文件和派生索引服务于检索、上下文装配和 UI 展示。它们不能成为作品事实的唯一来源,也不能被 Agent 静默改写成作者意图。

当外部编辑器改动文件时,系统以文件事实为准,并让相关 pending 审批失效。系统不尝试自动合并作者未确认的冲突。

## 失败语义

- 文件写入失败:审批不能标记为已生效。
- 索引刷新失败:作品文件保持已写入,相关检索能力进入降级或待修复状态。
- 外部编辑冲突:相关审批失效,用户重新审定。
- 连接或 native binding 异常:阻断写入路径,不得落入半写状态。

## 用户可见结果

用户看到的是作品文件、设定资料、审批是否生效、以及需要处理的冲突。用户不需要理解索引表如何更新,但系统必须能说明某个审批为什么失效或某个检索为什么暂不可用。
