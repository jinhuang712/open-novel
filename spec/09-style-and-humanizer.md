# 09 · Style And Humanizer

本文档定义去 AI 味、人味改写和风格守恒的技术边界。读完本篇应能理解:Humanizer 是表达层改写能力,不是剧情修改能力;它如何使用用户范文和经验;以及改写失败时为什么宁可保留原文也不能越过事实。

## 要解决的问题

AI 初稿常见问题是句式机械、情绪不贴、网感不足或和作者手感不一致。Humanizer 解决的是“表达像不像这本书/这个作者”,不是“剧情应该怎么发展”。

因此 Humanizer 只能在已明确的事实边界内改写表达。

## 主权对象

Style And Humanizer 拥有:

- 去 AI 味请求。
- 风格约束。
- 用户范文特征。
- 改写前后差异说明。
- 表达层风险提示。
- Humanizer 输出的审批边界。

它不拥有角色设定、剧情结果、世界规则、守则风险和章节主流程。

## 输入

Humanizer 的输入必须包含:

- 待改写文本。
- 不能改变的事实清单。
- 当前章节语境。
- 用户风格偏好或范文特征。
- 当前模式和审批要求。
- 已知守则风险。

如果事实清单不完整,Humanizer 不能对高风险正文做自动改写。

## 改写范围

允许改:

- 句式节奏。
- 口语感和网感。
- 情绪表达。
- 描写密度。
- 语气和叙述贴合度。

不允许改:

- 人物做过什么。
- 人物关系。
- 能力边界。
- 世界规则。
- 伏笔和承诺。
- 章节结局。

一旦改写触碰事实,结果必须作废或进入人工确认,不能作为普通润色通过。

## 用户风格与经验

风格来源包括用户范文、历史采纳改写、Settings 中的风格偏好和 Reflector 提炼的经验。它们只影响表达方式,不能覆盖当前显式指令和项目事实。

当用户修改风格偏好时,后续 Humanizer 应使用新偏好;旧经验是否继续生效由 [02](./02-runtime-state.md) 和 [11](./11-settings-and-onboarding.md) 的经验管理语义决定。

## 主路径

```mermaid
flowchart LR
  Text[待润色文本] --> Facts[不可改事实]
  Facts --> Style[风格约束]
  Style --> Humanizer[Humanizer 改写]
  Humanizer --> Diff[差异说明]
  Diff --> Validate[事实/守则复核]
  Validate --> Approval[审批或直接展示]
```

Humanizer 输出仍然是 proposal。只有用户接受或在明确安全的展示路径中,改写才会替换作品内容。

## 失败语义

| 失败 | 系统行为 |
|---|---|
| 改写改变事实 | 结果作废或进入人工确认 |
| 风格漂移 | 保留原文,允许用户重试或调整风格 |
| 输出不可解析 | 不写入作品 |
| 范文不足 | 标记风格依据不足,不强行拟合 |
| 与当前指令冲突 | 当前指令优先 |
| 与守则冲突 | 进入 Creative Engine 风险处理 |

## 用户可见结果

用户看到改写建议、差异说明、风格依据和是否触碰风险。系统不把“更像人写的”作为绕过审批、守则和事实一致性的理由。
